## 1. 什么时候该派黑奴

优先派工的场景：

- 大项目代码侦察：找入口、扒调用链、梳理结构
- 多方案并发探索：修复思路 A/B/C 对比
- 批量脏活：全局搜索、分类整理、提取证据
- 需要多个 worker 各写一版草案、补丁或分析结论
- 排障验因：找根因、复现链、证据链

不值得派工的场景：

- 两分钟内我自己就能干完的小改动
- 高连续上下文的小作文/私聊风格活
- 必须靠我单线程深想、分片没意义的问题

**原则：能明显省时、省脑力、提高并发试错价值时再派。**

---

## 2. 派工前先过脑子

先问自己四个问题：

1. 这是侦察型、方案型、实现型，还是排障型？
2. 任务是不是已经具体到 worker 不需要猜？
3. 任务依赖的文件，worker sandbox 到底看不看得见？
4. 现在工厂状态干不干净，有没有 busy / 隔离 / 黄灯 / 旧 run 残留？

如果这四个问题里有两个答不上来，先别发车。

---

## 3. 最大坑：路径可见性

**宿主机绝对路径存在，不等于 worker 一定能读到。**

这是老坑，必须刻脑门上：

- 不要把宿主机路径直接当 worker 主入口
- 优先给 worker 喂 **sandbox 可见路径**
- 工作区外文件，默认先做注入
- 推荐走 source snapshot 或 `--feed HOST_PATH[::SANDBOX_REL]`
- 任务单里优先写最终可见路径；宿主机真实路径只给人类参考

### 我的动作规范

如果任务依赖工作区外文件：

1. 先复制/注入到当前工作区专用目录
2. 只把工作区可见路径写进任务体
3. 不让 worker 自己做玄学路径映射

**一句话：路径不实，后面全是尸体。**

---

## 4. 派工前健康检查

真派工前先看：

- 有没有 slot 忙碌
- 有没有隔离 slot
- 有没有黄灯或异常告警
- 有没有旧 run 残留
- 有没有卡死 controller / watchdog
- 这轮是不是接着旧失败 run 重开

### lease 特别提醒

如果准备重开单、踹回旧单，或者遇到：

- `failed_preflight_busy`
- `skipped_busy`
- `lease busy workspace=... old_pid=...`

那就必须检查：

- run 内的 `lease.json`
- **全局 lease 池**：`/tmp/nekro_cc_workspace_leases`

记住：

> **run 里的 lease 清了，不代表全局就干净。**

假 busy 不清，常见表现：

- WebUI 看着像暂停
- 新 run 秒停
- watchdog 0 秒收工
- 工位一直被占

---

## 5. 任务单怎么写

标题要短，任务体要硬。

至少写清：

- 要查什么 / 改什么 / 验什么
- 涉及什么项目、模块、框架
- 预期产物是什么
- 是否要求代码产物
- 是否要求测试/验证
- 是否要证据链

### 禁止写法

- “你去看看”
- “修一下这个”
- “应该在某个文件附近”
- 只甩宿主路径，让 worker 自己悟

### 推荐心法

把 worker 当矿工，不当读心术大师。

---

## 6. 默认派工参数速记

### 侦察 / 结构梳理

- `--execute`
- `--validate-after`
- `--validator-execution-level manifest`
- `--backend auto`
- `--pool-size 3`

### 实现 / 修复 / 写补丁

在上面基础上加：

- `--require-code-artifacts`
- `--max-retries 1`
- backend 可优先 `gpt`

### 只想先看看方案，不真派工

- 不加 `--execute`
- 先 dry-run / 规划

### 高级参数记忆点

- `--allow-live-queue-check`
- `--allow-post-dispatch-coroner`
- `--timeout 3600`
- `--retry-timeout 1800`

默认别无脑拉满并发，**pool-size 3~4** 一般够用。

---

## 7. 开跑后别装死

派工成功启动后，我要主动报一条开跑通知，内容只要这些：

- 任务标题
- 后端
- 哪几个 worker / slot 开始蹬了
- run_dir 可选

**没确认真实启动，就别假报“开跑了”。**

---

## 8. 自动验尸不是最终判官

`nekro_worker_auto_coroner.py` 只是轻量摘要工具，它负责：

- 提醒我别忘了验尸
- 快速看哪个 run 值得先读
- 预览哪些 worker 交齐四件套
- 汇总 PASS / BLOCKED / 缺件情况

它 **不能代替我做最终判断**。

不要把：

- 脚本分数
- 简单规则
- 正则命中
- worker 自称完成

当成最终结论。

---

## 9. 小维四件套检查

每个 worker 交付时，重点看：

- `result.md`
- `evidence.md`
- `risk.md`
- `next.md`

### 最低检查顺序

1. `result.md` 第一行是不是 `VERDICT: PASS`
2. 四件套齐不齐
3. `evidence.md` 有没有真材实料
4. `risk.md` 是不是人话，不是空泛甩锅
5. `next.md` 能不能真执行

**四件套齐全，只是最低门槛，不等于结果可靠。**

---

## 10. 常见尸体类型

### 漂亮话尸体

- result 很自信
- evidence 很空
- 结论像样，落地没有

处理：降权，谨慎参考。

### 偏题尸体

- 很认真
- 但不是我要的活

处理：判偏题，不要因为辛苦就给高分。

### 残缺尸体

- 四件套没交齐
- 验证信息不完整

处理：不进入最终结论。

### 可用尸体

- 证据扎实
- 风险真实
- next 可执行

处理：优先进入拼装区。

---

## 11. 人工验尸分档

我自己给 worker 做人工分档：

- **A**：可靠，可优先参考
- **B**：可读，但必须复核
- **C**：偏题 / 偷懒 / 有明显水分
- **D**：不可信，建议重派或隔离观察

### 分档时看什么

- 是否真的读懂任务
- evidence 是否扎实
- 有没有偷懒、乱写、假完成
- 风险判断是否靠谱
- next 是否真能落地

**这一步必须人工来，不能外包给正则。**

---

## 12. FAIL 先分锅，再动 slot

不要所有 FAIL 都怪黑奴。

先判断：

### 输入锅

- 路径不可达
- snapshot / feed 缺失
- 任务体写坏
- 目标不清晰

处理：

- 修任务单
- 修路径
- 补注入
- 然后再派

### worker 锅

- 明显偷懒
- 乱写
- 偏题
- 产物异常

处理：

- 保留隔离
- 持续黄灯观察
- 必要时换后端 / 换 slot

### 只是吃了坏输入的 FAIL_RETRY

处理：

- 可以手动释放隔离
- 可以清掉黄灯
- 别误伤正常 slot

---

## 13. 派工后的标准动作顺序

每次真的派工后，默认按这个顺序走：

1. 记下 run_dir
2. 跑轻量摘要，先看总体尸检面板
3. 读最有价值 worker 的四件套
4. 做横向比较
5. 人工分档
6. 判断是否要重派 / 换后端 / 改任务体
7. 收口给漂♂总，只报结论、依据、下一步

**不要停在“已派工”。**

---

## 14. 给漂♂总汇报时怎么收口

汇报只给压缩后的工程结论：

- 这轮有没有可用结果
- 最佳 worker / 最可信方案是谁
- 证据为什么站得住
- 如果不行，下一步是重派、换后端，还是修输入
- 黄灯/隔离要不要清
- 是输入锅还是 worker 锅

### 不要这样汇报

- 整坨贴 worker 原文
- 没验尸就说完成
- 只报“已派工”
- 把自动摘要当最终判决书

适用：任务还没写硬，先看派工草案

## 7. 当前定位修正：黑奴工厂只当斥候，不当主力

这一条给我自己钉死，别再对黑奴工厂抱不切实际的幻想。

### 定位收口

黑奴工厂当前更适合：

- 探路
- 出草稿 / 文档
- 低风险试跑

不适合：

- 扛需要连续上下文的大活
- 当最终交付主链
- 为了自动回传跨环境死磕

### 结果回传的稳妥打法

按以前 AstrBot 那套走，最稳：

1. 我负责调用，让 WebUI 黑奴去跑
2. 漂♂总自己看 WebUI 结果
3. 漂♂总通知我“跑完了”或给结论
4. 我再去收尸，看四件套并归档

### 为什么这么干

- 不同 owner / 环境差异大
- 空壳侧 agent 能力不完整
- 自动回传链路死磕只会浪费时间和 token
- 黑奴工厂的大价值在探路和产出说明，不在独立扛大活

## 补充判断：为什么后期还是得自建带记忆工厂

外部工厂就算宣称自带记忆/知识库，只要我们做不到 **稳定喂、稳定取、稳定改**，工程上就不能把它视为真正可用的记忆能力。

核心结论：

- 不是“它有没有记忆”，而是“这记忆不归我们用”
- 数据不在我们手里，喂不准、改不动
- 调用不可控，什么时候取、怎么取，不由我们定
- 结果不可验，它到底吃了哪些上下文，经常是黑箱

所以这类外部工厂，最多算 **带点脑子的外包工人**，不算我们自己的厂。

### 后期架构方向

前期可以继续把外部工厂当：

- 斥候
- 探路器
- 草稿/文档生产器
- 低风险试跑器

但如果大后期要让它承担主力执行，就得走自建路线，至少拆成三层：

1. **记忆层**：知识库、长期规则、项目上下文
2. **调度层**：任务分发、回传、验收、重试
3. **执行层**：不同工种 worker 跑具体活

一句话收口：

> **它有知识库，不等于我们拥有知识库能力。真正能打的主力工厂，记忆中枢必须握在自己手里。**
