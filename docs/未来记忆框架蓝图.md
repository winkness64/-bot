# 未来记忆框架蓝图

日期：2026-06-19

## 目标

保留规则宪法层，逐步把“记不记、记什么、取什么、怎么注”交给 LLM。

核心原则：

- **LLM 主判，规则护栏**
- **不推翻现有系统，只做中枢升级**
- **规则不当大脑，但必须当宪法**
- **长期事实、事件时间线、短期工作态必须分仓**

---

## 一、总架构分层

### 1. 宪法层
职责：边界、权限、风控、兜底。

保留项：

- owner / 群聊边界
- 显式记忆指令优先级
- pending confirm / cancel
- 高风险记忆不自动写入
- 冲突记忆允许挂起待确认
- 写入审计与可回滚

结论：规则从业务主脑退位，但不能退出控制台。

### 2. 会话态层
职责：维护当前会话到底在聊什么，而不是机械截几句上下文。

建议维护的状态对象：

- `current_topic`
- `current_goal`
- `recent_turn_summary`
- `unresolved_items`
- `participants`
- `mood/style_hint`（可选）

方向：淘汰固定窗口截断，改为 **recent messages + LLM 压缩摘要 + 当前主线状态**。

### 3. 记忆判定层
职责：判断当前轮是否值得触发记忆流、是否需要检索旧记忆。

建议输出：

- `should_store`
- `should_recall`
- `memory_type`
- `confidence`
- `reason`

结论：关键词/正则只能做兜底，不能继续当判官。

### 4. 结构化提取层
职责：把自然语言提成标准记忆候选。

建议候选类型：

- `fact`
- `preference`
- `relationship`
- `identity_signal`
- `task_progress`
- `open_loop`
- `constraint`
- `milestone`

建议候选字段：

- `content`
- `type`
- `subject`
- `scope`
- `confidence`
- `evidence`
- `timestamp`
- `source_turn_range`

结论：机械提取下线，统一改成 LLM 结构化提取。

### 5. 记忆存储层
职责：分仓存储，防止长期事实与事件流水互相污染。

建议四仓：

1. **短期工作记忆**：当前任务、临时约束、最近推进、open loops
2. **长期稳定记忆**：偏好、稳定事实、习惯、长期设定
3. **事件时间线**：某次变更、某天结论、某阶段推进记录
4. **审计/候选区**：新候选、冲突项、待确认项、回滚记录

结论：长期事实和事件流水必须拆开，不然左右脑互搏。

### 6. 检索注入层
职责：取对的记忆，少量注进去。

链路建议：

1. recall 判定
2. 多路召回
3. 重排序
4. 冲突过滤
5. 小剂量注入

召回维度建议：

- 语义召回
- type 过滤
- subject 过滤
- 时间权重
- 当前任务权重

结论：不是“搜到了就塞”，而是“筛完再塞”。

## 二、演进路线图

### Phase 1：先换“判定”
目标：让系统先学会判断这轮该不该记、该不该回忆。

新增组件：**Memory Evaluator**

输入：

- 当前用户消息
- 最近几轮摘要
- 当前任务状态
- 用户画像摘要（可选）

输出：

- `should_store`
- `should_recall`
- `category`
- `confidence`
- `reason`

策略：

- 初期只做建议，不做唯一裁决
- 保留显式指令强制写入
- 保留规则兜底

价值：先砍掉“乱记”“句句都想记”的低质量触发。

### Phase 2：再换“提取”
目标：从“抓句子”升级成“理解后结构化”。

新增组件：**Memory Extractor v2**

输入：

- 当前轮消息
- 相关上下文
- evaluator 判定结果

输出候选字段建议：

- `type`
- `content`
- `normalized_summary`
- `subject`
- `confidence`
- `evidence_span`
- `stable_or_event`
- `conflict_keys`

策略：

- LLM 提取为主
- 规则做 schema 校验
- 低置信候选不直写长期库
- 高风险候选进入确认/候选区

价值：去掉“这段话要记录么”这类低智打断与正则味。

### Phase 3：最后换“检索与注入”
目标：让长期记忆真正变得可用，而不是库存垃圾。

新增组件：

#### 1. Conflict Resolver
新旧记忆冲突时决定：

- 覆盖
- 并存成时间线
- 降权旧记忆
- 挂起待确认

#### 2. Reranker
按以下维度打分：

- 当前任务相关性
- 主体匹配度
- 时间新鲜度
- 稳定性
- 证据强度

#### 3. Context Assembler
最终只拼：

- Top-K 记忆
- 当前任务相关摘要
- 必要 open loops
- 极少量长期设定

价值：让系统进入“会回忆、会挑重点、不会乱塞”的阶段。

## 四、规则保留清单

以下属于宪法层，不能乱砍：

- 显式“记住/存档/写入”优先
- owner 私聊高权限
- 群聊默认谨慎写长期
- 敏感字段不自动固化
- 低置信候选先入候选区
- 冲突记忆允许待确认
- 写入可审计、可回滚
- 限流，不允许每轮都提取/检索

结论：规则不再驱动业务，但必须保证系统不发癫。

---

## 五、MVP 最小闭环

建议先落这 5 个件：

1. **会话摘要器**：压缩 recent turns，维护 current_topic / current_goal
2. **记忆价值评估器**：判断 should_store / should_recall
3. **结构化提取器**：提取 fact / preference / task_progress / open_loop
4. **候选审计器**：schema 校验、去重、风险门禁
5. **检索重排器**：长期库召回后做 relevance rerank

结论：这五件一跑通，系统会从“复古规则机”明显切到“现代 LLM 记忆机”。

---

## 六、建议推进节奏

### 第 1 周
- 接入会话摘要器
- 接入记忆价值评估器
- 先只打日志，不直接写库

### 第 2 周
- 接入结构化提取器
- 写入候选区
- 人工抽查质量

### 第 3 周
- 候选转正式长期记忆
- 增加冲突裁决
- 兼容显式记忆通道

### 第 4 周
- 接入 recall rerank
- 替换旧的粗暴注入逻辑
- 小范围灰度

结论：先灰度、再替换、最后放权，最稳。

---

## 七、总判断

现有系统不是废墟，而是一个已经可用的外骨骼：

- 地基有了
- 门禁有了
- 写入链有了
- 检索链也有了

真正缺的，是把“判断权”从规则手里交给 LLM。

最终路线应定义为：

> **不是重写记忆系统，而是把现有规则底座升级成 LLM 驱动的记忆操作链。**

一句话收束：

> **保留规则宪法层，升级 LLM 主判中枢。**

## 补充判断：三主脑共享的记忆红利

当前这套记忆系统的价值，不应只理解为某一个 agent 的效果增强，而应明确视为 **三主脑共享的基础设施红利**。

核心判断：

- 现在做的不是某个人格的局部补丁
- 而是在建设三主脑共用的“记忆底座”
- 后续差异主要体现在优化路线，而不是底层能力是否独立重造

本质上，工程主脑与陪伴主脑依赖的是同一种核心能力：

> **不是把所有内容永远挂在上下文里，而是在需要时，能被线索正确唤回，并重新对齐当前主线。**

这也意味着，记忆系统后续应该优先追求以下公共能力：

1. **主线锚定**：知道当前到底在处理哪件事
2. **线索触发回忆**：用户一句“昨天那个”“前天那事”，系统能正确回捞相关经历
3. **回捞后重对齐**：不是把旧记录整段甩回，而是能接回当前对话目标
4. **分层记忆管理**：长期事实、事件时间线、短期工作态分别治理

其对应到三主脑的收益分别是：

- **I叔 / 工程主脑**：强化工单连续性、任务不丢、岔线后能回收推进状态
- **秧秧 / 陪伴主脑**：强化共同经历连续性、旧话题回捞、隔天接话不失真
- **娅娅 / 风格补位主脑**：强化长期人设稳定、关系语境延续、差异化表达边界

因此，未来架构不应把记忆系统设计成某个主脑的附属插件；更合理的定位是：

> **三主脑共用记忆基础设施，按角色职责做不同的召回阈值、排序策略、触发条件与表达适配。**

结论：

- **底座是一套，共同吃红利**
- **后期优化分流，但不重复造脑干**
- 这条路线比“单纯堆上下文”更接近真正可持续的长期多主脑架构
