# README_PRIVATE_CONTEXT_TOOL_TIMEOUT_REMEDIATION_PLAN_20260617.md

## 1. 背景
漂♂总反馈当前私聊链路存在“偶发断片”体感，主要表现为：
- 长对话过程中遗忘前文约束或已确认结论；
- 工具多步任务中，后续收口阶段出现上下文漂移；
- 模型超时后，回答收口质量下降，偶发整轮语义重置；
- 先前已有一次优化，症状有所减轻，但未根治。

当前已确认的 private 模型链：
- active: `gpt_5_4`
- explicit fallback: `gpt_5_4_mini -> v4_pro -> gpt_5_5`

本施工单只给方案，不直接修改线上逻辑。

---

## 2. 问题拆解
本问题不视为单点故障，按四层处理：

### 2.1 上下文注入层
疑似风险：
- 近轮对话注入条数偏少；
- 上下文预算偏紧，工具结果挤压有效对话；
- 用户硬规则、当前任务目标、临时确认结论未分层保留；
- 普通裁剪策略按“最近文本”工作，未按“任务重要性”工作。

### 2.2 短期记忆层
疑似风险：
- 短期上下文仅依赖进程内缓存；
- 重启后短期状态无法恢复；
- 当前任务锚点缺乏 rolling summary；
- 先前对话中的关键限制条件可能在多轮后被自然挤出。

### 2.3 工具调用层
疑似风险：
- 工具结果原文过长回灌，污染后续上下文窗口；
- 多步任务缺少阶段性任务摘要，导致“查着查着忘了为什么查”；
- 工具完成后的收口 prompt 未显式携带“当前目标/已知结论/待回答项”。

### 2.4 模型超时与回退层
疑似风险：
- 不同任务复杂度共用同一超时思路；
- 首轮生成、工具后收口、长文交付未差异化处理；
- 超时后 fallback 虽存在，但触发时机和语义续答策略可能不理想；
- 部分轮次可能发生“还没彻底失败，但已经丢掉前序推理上下文”的问题。

---

## 3. 本次改造目标
1. 提高私聊多轮任务中的上下文稳定性；
2. 降低工具链路对主上下文窗口的污染；
3. 优化超时与 fallback 行为，减少“整轮失忆式失败”；
4. 全过程可观测、可灰度、可回滚；
5. 优先做保守增量，不直接大改骨架。

---

## 4. 总体策略
执行原则：**先观测，后收敛；先分层，后扩容；先保守灰度，后结构升级。**

### 阶段 A：补观测
先确认断片发生在哪一层，而不是直接扩大上下文。

### 阶段 B：稳上下文
将“最近消息堆叠”改为“最近对话 + 任务摘要 + 用户硬规则”三层结构。

### 阶段 C：瘦工具回灌
限制工具原文进入主 prompt，只回灌结论、状态、异常、下一步所需字段。

### 阶段 D：调超时与回退
按轮次类型区分策略，减少无意义等待与失败后整轮报废。

### 阶段 E：做可恢复短期记忆
必要时增加轻量落盘恢复，解决重启后短期断片。

---

## 5. 具体施工项

## 5.1 施工项 A：增加最小可用观测
### 目的
定位断片根因，避免盲改。

### 需要记录的观测字段
每轮请求至少记录：
- 会话标识；
- 轮次序号；
- 实际使用模型 profile；
- 是否触发 fallback；
- 请求阶段类型：首轮 / 工具后收口 / 长文交付；
- 注入的上下文块类型与长度；
- 被裁剪块数量与裁剪后长度；
- 工具结果回灌长度；
- 是否发生 timeout；
- 最终响应是否完成。

### 输出要求
- 默认写入 debug/diagnostic 日志；
- 正常前台不展示；
- 日志中不暴露 key/token/base_url；
- 仅记录结构化长度、类别、状态，不落敏感正文。

### 验收标准
可以回答以下问题：
- 断片发生在哪类轮次最多；
- 断片前是否发生异常裁剪；
- 断片是否与 fallback/timeout 强相关；
- 工具结果是否显著挤占 prompt 预算。

---

## 5.2 施工项 B：上下文分层注入
### 目标结构
建议将注入内容拆为四层：

1. **用户硬规则层**
   - 如称呼规则、输出约束、权限语境；
   - 小体积、强常驻；
   - 不参与普通最近轮次裁剪。

2. **当前任务摘要层**
   - 当前在解决什么；
   - 已确认事实；
   - 已排除结论；
   - 待完成交付项；
   - 每轮更新，长度严格限制。

3. **最近对话层**
   - 近 12~16 条有效消息；
   - 优先保留 owner 明确结论、指令、确认；
   - 降低寒暄/重复确认的优先级。

4. **工具结论层**
   - 只保留必要摘要；
   - 不直接灌入长 stdout/stderr/log 原文。

### 实施建议
- 不先无脑增大全局字符预算；
- 优先按层级分配固定预算；
- 若预算紧张，优先裁掉工具原文和无效寒暄。

### 验收标准
同一任务连续 10+ 轮后，仍可保持：
- 用户称呼不漂移；
- 当前任务目标不漂移；
- 已确认模型链/事实不被后文覆盖；
- 工具结论能被正确引用。

---

## 5.3 施工项 C：引入 rolling summary
### 目标
解决“最近消息保住了，但任务主线丢了”的问题。

### 摘要内容
每个活跃私聊会话维护一个短摘要，建议包含：
- 当前任务一句话；
- 已确认关键信息 3~5 条；
- 当前待办 1~3 条；
- 禁止事项/不能改动项；
- 最近一次 owner 明确决策。

### 更新时机
- owner 明确确认后更新；
- 进入多步工具任务前更新；
- 工具任务阶段完成后更新；
- 长文交付前更新。

### 约束
- 长度上限固定；
- 只保留事实与决策，不写情绪化废话；
- 摘要更新失败时，不影响主流程。

### 验收标准
即使最近消息中没有完整上下文，仅靠摘要 + 近轮消息也能复原当前任务。

---

## 5.4 施工项 D：工具结果瘦身回灌
### 目标
减轻工具输出对 prompt 的污染。

### 改造原则
工具结果进入模型前，先压成结构化摘要：
- status: 成功 / 失败 / 部分成功；
- object: 查了什么对象；
- key findings: 关键结论；
- errors: error/exception/traceback 摘要；
- next step hints: 下一步可执行动作。

### 不建议直接回灌的内容
- 大段日志原文；
- 超长目录列表；
- 无关 stdout；
- 重复状态输出。

### 多步任务补强
在工具第 N 步后，插入一段自动任务锚点：
- 当前目标；
- 已完成步骤；
- 当前发现；
- 下一步原因。

### 验收标准
工具步数增加时，前台回答长度可略短，但目标不跑偏，结论不失忆。

---

## 5.5 施工项 E：超时与 fallback 策略细化
### 目标
减少“卡住—超时—换模型—上下文断裂”的连锁反应。

### 建议策略
按请求类型分层：

1. **短回复/普通对话**
   - 维持较快时限；
   - 超时可直接快速 fallback。

2. **工具后收口**
   - 适度放宽时限；
   - 强制带上当前任务摘要；
   - 若超时，fallback 继承同一任务摘要再答。

3. **长文交付/施工单/总结**
   - 使用更长时限；
   - 必要时允许两段式生成：先骨架，后正文；
   - 超时后优先保留已完成骨架，不让整轮作废。

### fallback 原则
- fallback 不是只换模型，还要带着“同一份任务锚点”继续；
- 触发 fallback 时记录原因：timeout / api error / policy-like refusal / empty output；
- 若 fallback 成功，应能标记为恢复性成功，便于后续统计。

### 验收标准
超时存在时，用户体感从“整轮失忆”下降为“回复略慢但主线还在”。

---

## 5.6 施工项 F：短期上下文轻量落盘恢复
### 目标
避免进程重启后彻底失去任务主线。

### 方案建议
仅落盘以下轻量信息：
- 会话级 rolling summary；
- 最近少量关键轮次索引；
- 最近 owner 明确决策；
- 恢复时间戳。

### 注意事项
- 不把整段聊天历史暴力持久化为主路径；
- 不和长期记忆混用；
- 恢复失败时允许静默降级。

### 验收标准
服务重启后，同一 owner 私聊继续施工任务时，主线不至于完全断裂。

---

## 6. 推荐实施顺序
建议顺序如下：

1. 施工项 A：补观测
2. 施工项 B：上下文分层注入
3. 施工项 C：rolling summary
4. 施工项 D：工具结果瘦身回灌
5. 施工项 E：超时与 fallback 策略细化
6. 施工项 F：轻量落盘恢复

原因：
- 先观测，避免误判；
- 先改注入结构，再改超时，否则可能只是把垃圾上下文喂得更多；
- 落盘恢复放最后，避免先把坏摘要固化下来。

---

## 7. 风险评估
### 风险 1：上下文变长导致整体更慢
应对：
- 先分层再扩容；
- 对每层设置硬上限；
- 优先压缩工具结果。

### 风险 2：摘要写错后稳定传播错误结论
应对：
- 摘要只在明确确认点更新；
- 对“推测”与“已确认”做标签区分；
- 必要时允许 owner 决策覆盖摘要。

### 风险 3：fallback 变多导致成本升高
应对：
- 只在超时/空输出/关键失败时触发；
- 将恢复性成功单独统计，评估真实收益。

### 风险 4：落盘恢复污染长期记忆边界
应对：
- 严格限定为短期任务缓存；
- 不写入长期记忆主库；
- 可单独开关。

---

## 8. 回滚方案
本改造必须支持逐项回滚。

### 回滚要求
- 每个施工项带独立开关或独立提交；
- 修改前先保留配置与相关代码冷备；
- 日志观测功能可单独关闭；
- 摘要注入、工具瘦身、超时策略、落盘恢复均可分别撤销。

### 回滚顺序建议
若上线后异常：
1. 先关落盘恢复；
2. 再关超时/fallback 新策略；
3. 再关工具结果瘦身改动；
4. 保留观测日志用于继续定位；
5. 最后再撤上下文分层注入。

---

## 9. 验收用例
建议至少执行以下用例：

### 用例 A：长对话稳定性
- 连续 12~15 轮私聊任务；
- 中途多次确认事实与限制；
- 检查末轮是否仍能正确引用前文。

### 用例 B：多工具串行任务
- 连续执行目录检查、读文件、状态判断、总结输出；
- 检查收口时是否忘记最初目标。

### 用例 C：超时恢复
- 模拟长文生成或人工压低超时；
- 验证 fallback 后是否仍沿用同一任务摘要。

### 用例 D：重启恢复
- 在任务进行中重启服务；
- 检查是否能恢复当前任务摘要与最近决策。

### 用例 E：错误摘要防扩散
- 人工制造一条未确认推测；
- 检查后续摘要是否把推测误写成既定事实。

---

## 10. 预期收益
若按上述顺序实施，预期收益为：
- 私聊多轮任务断片率明显下降；
- 工具多步任务收口更稳；
- timeout/fallback 不再高概率带来语义重置；
- 重启后的任务延续性增强；
- 后续若仍有问题，也能从观测日志快速定位层级。

---

## 11. 不建议本轮立即做的事
以下动作本轮不建议直接上：
- 直接大幅拉高全局 prompt 预算；
- 不带观测就更换 private 主模型；
- 先把整段聊天历史暴力持久化；
- 工具输出原文全量回灌；
- 不做开关直接重构整个 memory pipeline。

---

## 12. 下一步施工建议
下一步建议输出第二份配套文档：

**《模块级改动点清单 + 配置开关设计 + 灰度步骤》**

该文档应明确：
- 具体改哪些模块；
- 每个模块改动目的；
- 新增哪些配置项；
- 默认值建议；
- 灰度顺序；
- 回滚命令/回滚动作。

---

## 13. 结论
本问题的主因大概率不是单一模型能力不足，而是：
- 上下文注入结构不够抗多轮；
- 工具回灌过重；
- 短期记忆缺乏任务锚点；
- timeout/fallback 策略未按任务类型细分。

因此本轮建议走**保守增量修复路线**，先增强观测与上下文结构，再处理超时与恢复，不直接大换血。
