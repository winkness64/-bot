# WebUI / SSE 流式链路排查记录（2026-06-18）

## 背景
漂♂总要求排查：为什么 Nekro 那边 QQ 聊天经中转站会显示“流式”，而当前 yangyang_nonebot 这边不显示；并评估是否可在现有 8787 WebUI 链路上做文章，把 AstrBot / NoneBot 的内部通信流式能力复用起来。

## 本轮阶段结论

### 1. Nekro 的 OneBot11 适配器本身没有“消息编辑式真流”玄学
排查结论：
- QQ / OneBot V11 侧主要仍是普通发送消息。
- 未发现依赖 `edit_message`、单条消息增量覆盖、OneBot 特制 delta 流协议之类的实现。
- 因此，中转站显示“流式”并不说明 QQ 侧有特殊流式编辑能力。

### 2. Nekro 的“流式观感”核心来自 Web SSE 实时事件流，不是 QQ 适配器魔法
已确认 Nekro 存在面向聊天频道的 SSE 流接口，前端/中转站订阅该实时事件流后，可以呈现流式消息到达效果。
结论：
- QQ 侧：普通消息落地。
- Web / 中转站侧：通过 SSE 订阅实时聊天事件。
- 因此“中转站显示流式”的根因更接近 Web 实时事件推送，而非 OneBot11 适配器层。

### 3. 当前问题的关键差异，不在 QQ 分段发送，而在“中转站消费的流接口”
已基本排除“只要在 QQ 里分段发，中转站就一定认流”的想法。
更合理的判断是：
- 中转站判定流式，主要依赖 HTTP SSE / event-stream 或等价的实时事件接口。
- 如果上游只是一次性 JSON 响应，即便 QQ 侧另行分段发送，中转站也未必会显示为流式。

### 4. 8787 WebUI 现有代码里，AstrBot 链路已经是流式代理，可直接借力
漂♂总提示后继续排查，确认 8787 WebUI 里：
- 存在 AstrBot 的流式转发链路。
- 8787 对 AstrBot 上游使用 `Accept: text/event-stream`，会持续读取 AstrBot SSE 并向前端继续转发。
- 这说明 8787 的前端/中转逻辑本身具备“消费并转发流”的能力。

工程意义：
- 8787 并不是只能显示一次性结果。
- AstrBot 路由已经证明：这套 WebUI 管道可以承载真流式输出。

### 5. NoneBot 当前在 8787 内部链路中，还不是同等级“上游真流”
目前对本地 NoneBot 的代理链路，更像是：
- 8787 -> NoneBot：普通本地 JSON 请求
- 8787 -> 前端：再包装成 SSE 事件输出

也就是说：
- 前端这段看起来是 SSE；
- 但 NoneBot 上游目前并没有持续 chunk / delta 级别的流式接口；
- 因而现在更像“用 SSE 外壳包住一次性结果”，不是和 AstrBot 同级的真流代理。

## 当前最重要的工程判断
真正可做文章的位置，不是继续死磕 OneBot11 适配器，而是：

**复用 8787 现成的流式代理框架，给 NoneBot 增加一个真正的本地 SSE 输出接口。**

推荐方向：
1. 在 NoneBot 侧新增本地流式接口，例如 `/yy/api/chat/send_stream`
2. 响应类型为 `text/event-stream`
3. 在生成回复时，把 chunk / delta / 状态事件持续吐出
4. 8787 仿照当前 AstrBot 代理方式，持续读取该 SSE
5. 8787 再将事件转发给前端，形成真正的 WebUI 流式显示

## 为什么这条路线最顺
原因：
- 8787 前端已经能消费流
- 8787 后端已经会转发流
- AstrBot 链路已经是活样板
- 当前缺的不是 WebUI 能力，而是 NoneBot 上游缺少一个对口 SSE 源

## 现阶段可执行下一步
下一步应继续做两件事：

1. 细扒 8787 中 AstrBot 流代理的事件格式
   - 看它给前端发哪些 event / data
   - 识别最小兼容字段集合

2. 为 NoneBot 设计对口 SSE 接口
   - 尽量复用 AstrBot 已被 8787 接受的事件格式
   - 降低前端改造量
   - 优先做到“上游真流、8787 转流、前端直接显示流式”

## 追加结论（本轮纠偏，2026-06-18 15:02 CST）

### 6. 需要纠正的方向：目标不是“让 QQ 看起来流式”，而是“对上游模型请求真流式”
本轮进一步确认：
- 漂♂总此前要求的重点，不是 QQ 前台一定要呈现打字机式流式观感。
- 真正目标是：**对模型上游发起请求时走真 `stream=true`，从而骗过长响应等待、降低无输出超时风险，让后续整条链更稳。**
- 因此，QQ 是否显示流式、是否分段发送，只是体验层问题，不是本需求的核心验收标准。

### 7. Nekro 当年“骗 API 是流式”的核心，不在 OneBot11，也不在 anthropic 测试兼容文件
复核后确认：
- 先前拿到的 `anthropic.py` 并不是主战场，其中 openai fallback 甚至出现 `stream: false` 的兼容/探活式写法。
- Nekro 真正的主调用链核心在 **OpenAI 聊天生成路径**，其设计重点是：
  1. 对模型上游请求显式使用 `stream=true`
  2. 服务内部持续消费 chunk / delta，而不是一次性等待完整结果
  3. Web / 中转站再将这一实时过程以 SSE 方式继续向前转发
- 也就是说，Nekro 的“骗流式”本质是：**上游真流 + 中间持续消费 + Web SSE 转发**。

### 8. 8787 WebUI 为什么能直接流式调 NoneBot，而 QQ 不行
本轮收口结论：
- 8787 WebUI 能流式，是因为它本质上是 HTTP 客户端，可以：
  - 请求 `text/event-stream`
  - 保持长连接
  - 持续读取 SSE event / chunk
  - 读到一段就继续向前端转发
- QQ / OneBot11 不具备同等协议形态：
  - 它接收的是离散消息事件
  - 它发送的是 `send_private_msg` / `send_group_msg` 这类离散投递动作
  - 它不是一个订阅 SSE 再增量渲染单条消息的标准客户端
- 因此：
  - **WebUI 负责看流**
  - **QQ 负责收最终结果（或最多做普通分段发送）**
- 这也是为什么“8787 能流式调用 NoneBot”与“QQ 不能原样获得同等级流式观感”并不矛盾。

### 9. 当前真正还差的关键一步
截至本轮，最关键缺口已经收敛为一条：

**NoneBot 当前主生成链，还没有稳定实现“向模型上游发起真 `stream=true` 请求，并在本地持续消费 chunk，再通过本地 SSE 接口吐给 8787”。**

换言之，真正要补的是：
1. NoneBot -> 模型上游：真流式请求
2. NoneBot 内部：持续消费流
3. NoneBot -> 8787：提供真实 SSE 源（而非仅用 SSE 外壳包装一次性结果）
4. 8787 -> 前端：复用现有 AstrBot 式流转发链路

### 10. 当前建议的验收口径
后续若实现成功，验收重点应改为：
- 上游模型请求是否真实使用 `stream=true`
- NoneBot 是否真实按 chunk 持续消费，而不是等待完整响应
- `/yy/api/chat/send_stream` 是否持续输出真实 SSE 事件
- 8787 WebUI 是否能稳定看到连续事件流
- QQ 侧是否最终稳定收到结果（不强求消息编辑式流式观感）

### 11. 下一步排查/改造方向
建议下一步直接定位：
- yangyang / NoneBot 当前**实际发模型请求的主函数**在哪里
- 该主函数是否已支持 `stream=true`
- 若未支持，在哪一层补最稳：统一 LLM provider 层、聊天服务层，还是 Web 路由专用流式分支
- 同时确认 `/yy/api/chat/send_stream` 当前到底是：
  - 真正绑定上游 chunk 源
  - 还是仅把一次性结果重新包装成 SSE


## 追加结论（QQ ↔ WebUI 会话同步能力补充，2026-06-18 15:05 CST）

### 12. Nekro 更强的不是单纯“能流”，而是“同一会话可被 QQ 与 WebUI 同步观察/续写”
漂♂总补充的关键点非常重要：
- Nekro 的强项不只是把模型输出做成流式展示；
- 更关键的是，它能做到：
  1. **QQ 发来的消息，会同步出现在 WebUI 的同一会话视图里**；
  2. **WebUI 里补发的内部消息/继续对话，也会进入同一条会话上下文**；
  3. 两端看到的是**同一份会话状态**，而不是两个彼此独立的聊天窗口。

### 13. 这说明 Nekro 背后不是“QQ 和 Web 各聊各的”，而是有统一会话层/事件总线
从能力表现反推，其核心大概率包含三件事：
1. **统一会话ID / 线程ID**
   - QQ 来的消息不会只停留在适配器层；
   - 它会被映射进一个内部统一会话（conversation / session / thread）。
2. **统一消息落库或统一消息状态管理**
   - 用户消息、助手消息、内部消息，都进入同一个会话消息流；
   - WebUI 读取的是这条统一消息流，而不是单独读 QQ 历史。
3. **统一事件广播 / 订阅分发**
   - 当 QQ 侧产生新消息、助手开始回复、回复增量到达、回复结束时；
   - WebUI 订阅端会收到对应事件，因此能同步显示。

### 14. 因此，“QQ 信息同步 WebUI”本质上是会话总线问题，不只是 SSE 问题
需要明确区分两层：
- **SSE / 流式**：解决“事件如何实时往 WebUI 推”；
- **统一会话层**：解决“QQ 消息和 WebUI 消息为什么会落在同一条会话里”。

也就是说：
- 只有 SSE，没有统一会话层：只能做到 Web 看到流；
- 只有统一会话层，没有 SSE：能同步会话，但不一定实时；
- **两者都具备，才接近 Nekro 的完整效果。**

### 15. 对当前 yangyang / NoneBot 的工程启示
如果要逼近 Nekro 的能力，后续不能只做 `/yy/api/chat/send_stream` 真流化，还要补一层：

1. **会话统一映射**
   - QQ 私聊/群聊消息进入后，要映射到内部统一 conversation_id；
   - WebUI 打开该会话时，能读取同一条上下文。

2. **消息统一入流**
   - QQ 来的用户消息
   - WebUI 发的内部消息
   - 助手回复
   - 系统状态消息
   都应该进入同一个消息存储/事件流。

3. **事件统一广播**
   - 新用户消息
   - 助手开始回复
   - delta/chunk
   - reply done
   - error
   WebUI 应订阅这些事件并实时刷新。

4. **WebUI 反向发言也要走同一条会话管道**
   - 不是单独开一条“Web 专用聊天”；
   - 而是把 WebUI 发言作为同一 conversation 内的一条消息注入，再驱动后续回复。

### 16. 现阶段缺口应拆成两类，不要混为一谈
后续改造应分成两大块：

#### A. 流式链路缺口
- NoneBot -> 模型上游：真 `stream=true`
- NoneBot -> 8787：真 SSE
- 解决的是：上游稳定性、WebUI 实时输出

#### B. 会话同步缺口
- QQ 消息进入统一 conversation
- WebUI 读取同一 conversation
- WebUI 发言回注同一 conversation
- 解决的是：QQ / WebUI 双端同会话同步

### 17. 当前更准确的目标画像
最终要追的不是“QQ 也像 WebUI 一样显示流式”，而是：

- **QQ 是外部入口**
- **WebUI 是观测/介入面板**
- **两者共享同一会话状态**
- **WebUI 能实时看到 QQ 触发的回复过程**
- **WebUI 也能向该会话注入内部消息继续推进**

这才是 Nekro 那套体验真正牛逼的地方。

### 18. 后续排查/设计的优先顺序建议
建议后续按这个顺序推进：
1. 先确认当前 yangyang 是否已有内部统一 conversation/message store
2. 再确认 QQ 入站消息当前落在哪个会话抽象里
3. 再确认 WebUI 是否已有读取同会话消息流的接口
4. 最后再把 SSE 真流和会话事件广播绑到一起

这样才不会只做出“能流，但不同步”的半成品。

因此后续优先级调整为：

### P0：先搞定 QQ 链路背后的“真流来源”
当前最现实、收益最高的目标，不是短期抄完整会话同步神殿，而是先让当前 NoneBot 主生成链具备“骗上游按流式返回”的能力，让后续 WebUI / 8787 / 任务执行链都建立在稳定的流式输出基础上。

要点：
- 目标不是让 QQ 协议本身支持 SSE；

**短期先解决“骗到流”，长期再考虑“统一会话层”。**

但该方向属于中长期架构升级项，当前仅留档，不纳入本轮必须完成范围。
