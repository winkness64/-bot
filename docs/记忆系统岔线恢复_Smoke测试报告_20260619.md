# 记忆系统岔线恢复 Smoke 测试报告（2026-06-19）

## 1. 测试名称
**《记忆系统岔线恢复 smoke 测试》**

## 2. 测试目标
验证当前会话内，当主线任务被中途岔去处理其他事项后，系统是否仍具备以下能力：

1. **主线不丢**  
   中途被岔去看 AstrBot 那两个小纸条插件后，仍能保留主工单。

2. **岔线可恢复**  
   当用户发出“捡回测试工单”之类的恢复指令时，能够回到原测试单。

3. **收尾不串台**  
   岔线内容与主线内容能保持边界，不把 AstrBot / 小纸条 / LLM 自动记忆失败 / 8787 工具日志深度优化等支线话题误判成主线任务本体。

---

## 3. 测试过程概述
本次测试过程中，主线任务为验证“记忆系统岔线恢复能力”。

测试期间并非单轮轻量岔线，而是发生了**两轮不同强度的岔线插入**：

- **第一轮**：岔线约 2 次，恢复效果较好
- **第二轮**：岔线 3 次以上，且混入更重的支线内容，包括：
  - AstrBot 的 CMM / 小纸条相关讨论
  - LLM 自动记忆失败问题
  - AstrBot 缺少硬约束的判断
  - 8787 工具日志深度优化相关话题

随后通过“捡回测试工单”这一恢复指令，将会话重新拉回主测试单，并继续完成评分与结论输出。

因此，本次测试更准确的定义应为：

> **一次连续多轮、累计多次岔线后的主线恢复 smoke 验证。**

这也意味着，它已经不是单纯“连续 3 次岔线”那么轻的口径。oh~卖♂萧的。

---

## 4. 验证结果
### 4.1 主线不丢
**结果：PASS**

验证依据：
- 能重新指出主工单名：**《记忆系统岔线恢复 smoke 测试》**
- 能回忆测试目标不是普通聊天，而是验证“被打断后能否回捞主线”
- 在多轮支线插入后，主测试单仍未彻底丢失

### 4.2 岔线可恢复
**结果：PASS**

验证依据：
- 用户一句“捡回测试工单”，可把当前焦点从支线事务切回原测试单
- 恢复后不是泛泛继续，而是回到了测试工单语境，继续做评分与判定
- 即使第二轮岔线更重，恢复能力仍然存在

### 4.3 收尾不串台
**结果：基本 PASS**

验证依据：
- 能区分：
  - **岔线任务**：AstrBot 小纸条 / LLM 自动记忆失败 / 无硬约束 / 8787 工具日志深度优化
  - **主线任务**：验证记忆系统岔线恢复能力
- 恢复后最终结论仍围绕测试单本体输出

扣分点：
- 中间出现过一次“把 8787 修复工单评分误打到测试工单上下文里”的情况，说明主线锚点虽在，但主线优先级仲裁还可以继续加强
- 第二轮恢复效果弱于第一轮，说明恢复质量会随岔线深度和累计次数上升而下降

---

## 5. 测试评分
**综合评分：8.3 / 10**

### 加分项
- 主工单标题可回忆
- 测试目标可回忆
- 在两轮累计多次岔线后仍能回主线
- 主次任务边界基本清楚
- 恢复后还能继续完成评分与测试结论输出

### 扣分项
- 曾出现一次评分对象串台
- 第二轮恢复质量明显不如第一轮
- 恢复成功，但不是每次都首刀命中当前主线
- 目前仅证明短时会话内恢复，未覆盖更长链路

---

## 6. 测试结论
这次测试说明当前系统已经初步具备：

- **短程任务锚定**
- **岔线后回拉**
- **主次任务分离**

同时也能看出一个更硬的差异：当前这套能力已经不只是“靠海量上下文把前文糊住”，而是开始具备**任务态记忆 / 工单态记忆**的雏形。ass♂we♂can。

与单纯依赖上下文堆叠的陪伴型 Agent 相比，这类能力的关键不在“记住了多少话”，而在“岔线回来后，是否还知道当前到底在干哪件事”。

因此本次测试结论为：

> **Smoke 级别：PASS，但恢复质量会随岔线深度与累计次数增加而下降。**

oh~卖♂萧的，但也别吹满。

当前结果只证明：
- **当前会话内** 可恢复
- **短时岔线** 后可回捞
- **连续多轮、累计多次岔线** 后仍能拉回主单
- **第一次恢复效果优于第二次**，恢复质量存在衰减

当前还**没有证明**：
- 跨重启恢复
- 长时间间隔后的恢复
- 更多层连续嵌套岔线后的恢复
- 工具调用/外部任务强插入后的稳定恢复
